# agile-issue スキル

INVEST / SPIDR 原則に基づいてフィーチャーを分解し、GitHub Issue として登録するスキルです。

---

## 🎯 責務

- ユーザーが提示した「ゴール・現状・サイズ感」から Story / Task / Spike / Bug の各チケットを設計する。
- `references/` 配下のテンプレートを使って Issue 本文を生成する。
- GitHub MCP (`mcp_github-mcp-server_issue_write`) でリポジトリに一括登録する。
- ラベルを自動付与し、依存関係を本文にリンクして記載する。

---

## 📐 チケット種別と判断基準

| 種別 | ラベル | 判断基準 |
|------|--------|----------|
| **Story** | `story` | ユーザー価値が直接出るもの。「〜ができる」という形式で書ける |
| **Task** | `task` | エンジニア作業単位。インフラ整備・リファクタ・設定変更など |
| **Spike** | `spike` | 調査・検証。不確実性が高い・実装方法が未確定な箇所 |
| **Bug** | `bug` | 既存の動作が仕様と異なる |

---

## 🔬 INVEST チェックリスト（分割時に必ず適用）

- **I**ndependent — 他チケットへの依存が最小か？依存がある場合はリンクで明示する。
- **N**egotiable — 実装詳細は会話で変えられる余地を残す。
- **V**aluable — このチケット単独でユーザーまたは開発者に価値があるか？
- **E**stimable — 見積もれるサイズか？「調査中で不明」なら Spike に切り出す。
- **S**mall — 1日（≒600行差分以内 / 1PR）で終わるか？
- **T**estable — 完了条件（AC: Acceptance Criteria）が書けるか？

---

## ✂️ SPIDR 縦切り戦略

大きな機能を以下の軸で縦切りにする。

| 軸 | 説明 | 例 |
|----|------|----|
| **Spike** | 調査を先行させる | API仕様を確認するSpikeを切る |
| **Path** | 正常系→異常系の順に切る | まず正常ログイン、次にエラー処理 |
| **Interface** | UIとAPIを分離する | バックエンドAPIを先に完成させる |
| **Data** | 最小データセットから始める | 一部フィールドのみの最小フォームから |
| **Rules** | ビジネスルールを段階的に追加 | バリデーションを別チケット化 |

## 🛠️ 実行ワークフロー

### ステップ 1: ユーザーから3点を受け取る

以下の情報が揃っているか確認する。不足している場合は質問する。

```
【ゴール】     何を、誰が、どう使えるようになるか（1〜3行）
【現在の状況】 技術スタックと実装の進捗（1〜2行）
【分割ルール】 1チケットのサイズ感・正常系のみかどうか（省略可、デフォルト適用）
```

**デフォルトの分割ルール（省略時）:**
- 1チケット = 1PR = 1日以内
- 正常系のみ（異常系・バリデーションは別チケット）
- タイトルは「〜ができる」形式（Story）または「〜を〜する」形式（Task/Spike）

### ステップ 2: チケット設計（ドラフト提示）

AIが以下を実行する。

1. **依存関係グラフを頭の中で作成**し、実装順を決定する。
2. **Spike が必要な箇所を洗い出す**（不確実性 > 0.5 なら Spike に切る）。
3. `references/` のテンプレートを使い、**各チケットのドラフトを日本語で提示**する。
4. ユーザーに確認を取る（「このままGitHubに登録してOKですか？」）。

### ステップ 3: GitHub Issue への一括登録

ユーザーがOKを出したら `mcp_github-mcp-server_issue_write` を使って順次登録する。

```
- method: "create"
- owner: <リポジトリオーナー>
- repo: <リポジトリ名>
- title: <チケットタイトル>
- body: <テンプレートを埋めた本文>
- labels: ["story" | "task" | "spike" | "bug", "sprint:N"]
```

**登録順序:** Spike → Story（依存なし）→ Story（依存あり）→ Task

### ステップ 4: 登録完了の報告

登録した Issue のURL一覧をまとめてユーザーに報告する。

---

## 📁 テンプレートリファレンス

各チケット種別の Issue 本文テンプレートは `references/` 配下に格納されている。

- `references/story.md` — Story テンプレート
- `references/task.md` — Task テンプレート
- `references/spike.md` — Spike テンプレート
- `references/bug.md` — Bug テンプレート

---

## ⚠️ 注意事項

- Issue 登録前に必ずドラフトをユーザーに確認させること。無断登録禁止。
- ラベルが存在しない場合はユーザーに事前に作成を促す（またはラベルなしで登録）。
- 1回の登録チケット数は最大20件。それ以上は複数回に分けてスプリントを分割する。
- リポジトリ情報（owner/repo）はユーザーに確認すること。記録がある場合は既存情報を使用。
