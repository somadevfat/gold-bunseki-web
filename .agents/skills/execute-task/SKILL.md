# execute-task スキル

このスキルは、プロジェクトの設計・品質基準（カバレッジ100%、Lint/TSC厳守、ユニットテストでの網羅）に従って実装を遂行するためのワークフローです。無断での実装やPR作成を避け、各ステップでユーザーとの合意を取りながら安全に進めます。

## 🎯 責務

- GitHub Issueの情報取得と、それに基づいた実装計画の提案
- Gitブランチの管理（`develop`ブランチからの分岐）
- カバレッジ100%を徹底した実装と、定期的な静的エラー（Lint/Tsc）の排除
- 品質確認結果をまとめたPR準備用エビデンスの出力
- ユーザーに対する最終コードレビュー依頼（PR起票の承認待ち）

---

## 🛠️ 実行ワークフロー

### ステップ 1: Issue情報の取得と実装計画の提案
1. ユーザーから着手する **Issue番号** を受け取る。
2. GitHub MCP (`mcp_github-mcp-server_issue_read`) を用いて、Issueのコンテキストや要件を取得する。
3. 取得した情報から、**実装計画（対象ファイル、設計、テスト方針）** を作成し、ユーザーに提案する。
4. ユーザーに「この実装計画で進めてOKか、修正が必要か」を確認する。

*⚠️ここを勝手に飛ばして実装を始めてはならない。必ずユーザーの `OK` を待つこと。*

### ステップ 2: 作業ブランチの作成と切り替え
ユーザーから計画への `OK` が出たら、作業環境をクリーンにした上でブランチを切る。

```bash
git switch develop
git pull origin develop
git switch -c feature/issue-<Issue番号>
```

### ステップ 3: 作業開始・堅牢な実装とテスト
以下のルールに従ってコードを書く。

1. **Lint/TSCの随時解消**:
   実装中は随時リンター（`eslint` / `bun run lint:all` 等）およびコンパイラ（`npx tsc --noEmit`等）を実行し、見つけたエラーや警告はその場で直ちに修正する。
2. **ロジックのテストカバレッジ 100%**:
   `hooks`, `api`, `utils`, などのドメイン・アプリケーションロジック部分は必ずユニットテストを用意し、カバレッジ100%を達成すること。
3. **フロントの画面 (.tsx) はユニットテストで網羅**:
   UIコンポーネントやページファイルに対する表示・分岐の検証は、Testing Library と `bun test src/` によるユニットテストで担保する。

### ステップ 4: 作業完了報告とPR用エビデンス作成
すべての実装が完了したら、プロジェクト全体の最終クオリティゲートを実行する。

```bash
# フォーマットやLintのチェック
bun run lint:all

# 型チェックの通過
npx tsc --noEmit

# アプリケーションのビルドチェック
bun run build

# ユニットテストのカバレッジレポート出力
bun test --coverage

# フロントエンドのユニットテスト
cd apps/frontend && bun test src/
```

上記すべての結果（Passしたこと、カバレッジが100%であること等のコンソール出力）を収集し、**PRに添付するためのエビデンスドキュメント（Artifact、または簡潔なMarkdown）** としてユーザーに提示する。

### ステップ 5: ユーザーレビュー待ち（PR凍結）
- 完了の報告と共に、「コードの実装を完了しました。ユーザー側で確認・コードレビューをお願いします」として待機する。
- *⚠️ AI自身の判断で勝手にコミット・PushしてPRを上げてはならない。*
- ユーザーが「OK、PR作成して」と返答した場合にのみ、初めて `create-pr` スキルなどの後続ステップへと進む。
