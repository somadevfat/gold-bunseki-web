# ADR-0005: 考察ブログの記事ソースは型付きファイルベースから開始する

## ステータス
承認済み

## コンテキスト
考察ブログは現在 `apps/frontend/src/app/insights/page.tsx` 内の固定配列で表示している。今後、記事一覧、記事詳細、記事ごとのSEO、sitemap出力を追加するには、記事本文・slug・公開日・カテゴリ・メタ情報を単一のソースから取得できる必要がある。

候補は以下の3つ。

- Markdown/MDX: 記事をファイルとしてレビューでき、GitHub PRで運用しやすい。将来の本文表現も広げやすい。
- DB: 管理画面やユーザー投稿には向くが、初期実装では認証・編集UI・バックアップ運用が増える。
- Notion等の外部CMS: 執筆体験は良いが、ビルド・キャッシュ・API制限・外部サービス依存が増える。

このプロジェクトでは、ポートフォリオとして説明しやすい実装と、CIで検証できる小さな運用フローを優先する。

## 決定事項
初期実装では、考察ブログの記事ソースをリポジトリ管理の型付きファイルベースにする。

具体的には、Frontend内に `features/insights` ドメインを作り、記事データを以下の構造で扱う。

- `slug`: URLに使う一意な識別子
- `title`: 記事タイトル
- `description`: SEO description と一覧抜粋
- `category`: 表示カテゴリ
- `publishedAt`: 公開日
- `body`: 初期は構造化済みの段落配列として保持

初期の本文はTSオブジェクトの配列で実装し、MDXレンダラーやCMS連携は導入しない。UI側は `getInsightPosts`, `getInsightPostBySlug` のような読み取り関数だけに依存させ、将来MDX/DB/Notionに移す場合もページ側の変更を小さくする。

## 結果
メリット:

- 追加依存なしで #34 記事一覧、#35 記事詳細、#36 SEO/sitemap を段階的に実装できる。
- 記事追加・修正がPRレビュー対象になり、CIで型とテストを確認できる。
- `slug`、メタ情報、本文を同じソースで管理できる。
- 将来CMSへ移す場合も、読み取り関数の内側を差し替えやすい。

デメリット・リスク:

- 非エンジニア向けの編集体験はない。
- Markdown記法をそのまま書ける運用ではないため、長文記事が増えたらMDX化を再検討する。
- 公開予約や下書き管理は初期スコープ外になる。

再検討条件:

- 記事数が20本を超える。
- コード編集なしで記事を追加したい運用者が増える。
- 図表、埋め込み、コードブロックなど本文表現の要求が増える。
