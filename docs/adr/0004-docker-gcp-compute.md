# ADR-0004: GCP Compute Engine (VM) 採用と完全コンテナ化(バックエンド+DB)によるCI/CD基盤の構築

## ステータス
承認済み (Accepted)

## コンテキスト
ADR-0003にてバックエンドおよびデータベースのデプロイ先をVPSへ移行する方針を決定したが、具体的なホスティングプロバイダとアプリケーションの稼働形態について再検討を行った。
ポートフォリオとしての技術的アピールを最大化するため、「パブリッククラウド(GCP)の実践的なインフラ構築経験」と「Docker Composeを用いたバックエンド・DBの統合的なCI/CDパイプラインの構築」が極めて重要であると判断した。
1分ごとの高頻度バッチ処理等の要件がなく、リソース制約の範囲内で稼働させることが可能であるため、ネイティブ起動ではなくモダンな完全コンテナ化を選択する。

## 決定事項
以下のインフラストラクチャおよびデプロイメント構成を確定する。

1. **ホスティング**: `Google Cloud Platform (GCP) Compute Engine`
   - e2-micro インスタンス（Always Free枠の米国リージョン）を採用し、運用コストを最小化する。
   - GCPのVPCファイアウォール等を用い、インフラの自己管理を行う。
2. **デプロイメント構成（完全コンテナ化）**:
   - **Database**: `PostgreSQL 16` (Dockerコンテナ)
   - **Backend API**: `Bun.serve` (Dockerコンテナ)。`apps/backend/Dockerfile`を作成し、Bun公式イメージを用いてコンテナ化する。
   - 両者を `docker-compose.yml` に統合し、同一ネットワーク上で連携させる。
3. **CI/CD（将来展望）**:
   - GitHub Actions等からGCP VMに接続し、Docker Composeを用いて自動デプロイ（イメージ再ビルド・コンテナ再起動）を行う堅牢なパイプラインを構築する。

## 結果
- **メリット**:
  - バックエンドとDBの両方をDockerに載せることで、ローカル開発環境と本番環境（GCP VM）の差異が完全に無くなり、デプロイが安定する。
  - モダンなコンテナ技術（Docker）とパブリッククラウド（GCP）を用いたインフラ構築・CI/CD設計の経験をアピールできる。
- **リスク・デメリット**:
  - メモリ1GBの制約下（e2-micro）でDBとAPIの両方をコンテナ稼働させるため、極端なアクセス集中時や重いプロセスを実行した際にOOM (Out Of Memory) 発生のリスクがある。
  - 米国リージョンを採用するため、日本からのAPIリクエストに物理的なレイテンシ（100ms-200ms程度）が発生する。
